# ANC Lucky Wheel — เอกสารออกแบบฟีเจอร์รูเล็ตแจ้งงานใหม่

## 1. เป้าหมายและตัวชี้วัด

เป้าหมายของฟีเจอร์คือกระตุ้นให้พนักงาน (และบุคคลภายนอก) แจ้ง "งานใหม่" เข้ามาใน ANC Portal มากขึ้น จากปัจจุบันงานเข้าประมาณ 200 งาน/วัน เป็นงานใหม่ราว 20% (~40 งาน/วัน) ตั้งเป้าเพิ่มเป็น 30% (~60 งาน/วัน) และเติบโตต่อเนื่อง ภายใต้งบรางวัลไม่เกิน **5,000 บาท/วัน**

ตัวชี้วัดที่ควรติดตามหลังเปิดใช้: สัดส่วนงานใหม่ต่องานทั้งหมดรายวัน/รายสัปดาห์, จำนวนงานใหม่ที่ "ผ่านการตรวจสอบ" (ไม่ใช่แค่ที่กดแจ้ง), อัตรางานถูกตีตกเพราะซ้ำหรือไม่ใช่งานใหม่ (ตัวชี้วัดการโกง), และต้นทุนรางวัลต่องานใหม่ที่ใช้ได้จริง 1 งาน

## 2. กติกา (ตามที่เลือกไว้)

ผู้แจ้งงานจะได้สิทธิ์หมุนวงล้อ **หลังงานผ่านการตรวจสอบว่าเป็นงานใหม่จริง** (ไม่ใช่ตอนกดส่งฟอร์ม) — 1 งานที่ผ่านการตรวจ = 1 สิทธิ์หมุน สิทธิ์สะสมได้และหมุนเมื่อไหร่ก็ได้ภายใน 7 วัน หมุนแล้ว**ได้รางวัลเสมอ** ขั้นต่ำ 20 บาท รางวัลโอนเข้าบัญชีเป็นรอบ (แนะนำรายสัปดาห์) พร้อมประกาศรายชื่อผู้โชคดีเพื่อความโปร่งใสและช่วยโปรโมตแคมเปญไปในตัว

## 3. สัดส่วนรางวัลและการคุมงบ (ผ่านการรันซิมูเลชันแล้ว)

วงล้อโชว์ 8 ช่องขนาดเท่ากัน (20 / 50 / 100 / 200 / 300 / 500 / 700 / 1,000) แต่ความน่าจะเป็นจริงถ่วงน้ำหนักฝั่งเซิร์ฟเวอร์ดังนี้:

| รางวัล (บาท) | โอกาส | หมายเหตุ |
|---|---|---|
| 20 | 50% | รางวัลปลอบใจ ได้ทุกคน |
| 50 | 28% | |
| 100 | 14% | |
| 200 | 5% | |
| 300 | 2% | |
| 500 | 0.7% | |
| 700 | 0.3% | |
| 1,000 | 1.5% ต่อการหมุน แต่**จำกัด 1 รางวัล/วัน** | เซิร์ฟเวอร์ล็อกทันทีที่ออก |

ค่าเฉลี่ยต่อการหมุน (ไม่รวมแจ็คพอต) ≈ **59.6 บาท** ผลจากการจำลอง 5,000 วัน:

| งานใหม่/วัน | ค่าใช้จ่ายเฉลี่ย/วัน | เกินงบ 5,000? |
|---|---|---|
| 60 (เป้า 30%) | ~4,100 บาท | ไม่เคยเกิน |
| 80 | ~4,830 บาท | ไม่เคยเกิน |
| 100 | ~4,990 บาท | ไม่เคยเกิน |

กลไกคุมงบ (budget guard) ที่ทำให้ไม่เกินงบแน่นอน: ระบบเช็กงบคงเหลือก่อนสุ่มทุกครั้ง เมื่องบคงเหลือของวันต่ำกว่า 300 บาท วงล้อจะจ่ายเฉพาะ 20/50 บาท และไม่มีทางจ่ายรางวัลที่ใหญ่กว่างบคงเหลือ หากงบหมดจริง ๆ สิทธิ์หมุนจะ**ยกยอดไปวันถัดไป**แทนที่จะหมุนฟรีไม่ได้อะไร (สำคัญมาก — ถ้าตัดสิทธิ์ทิ้งคนจะหยุดแจ้งงานช่วงบ่าย/เย็น)

ข้อสังเกตจากซิมูเลชัน: ถ้าโครงการสำเร็จเกินคาด (แตะ 100 งาน/วัน) คนที่หมุนช่วงท้ายวันจะได้แต่รางวัลเล็กบ่อยขึ้น ควรพิจารณาเพิ่มงบหรือปรับน้ำหนักรางวัลลงทั้งกระดานเมื่อถึงจุดนั้น

## 4. Flow การทำงานของระบบ

ผู้ใช้แจ้งงานผ่านฟอร์มเดิมใน portal → ระบบ/แอดมินตรวจว่าเป็นงานใหม่จริงภายใน 24 ชม. → เมื่ออนุมัติ ระบบสร้าง spin token ผูกกับ job ID นั้น (1 งาน = 1 token, กันกดซ้ำด้วย unique constraint) → ผู้ใช้เข้าหน้า Lucky Wheel เห็นจำนวนสิทธิ์ → กดหมุน เซิร์ฟเวอร์สุ่มรางวัล บันทึกผลพร้อม log → หน้าเว็บเล่นแอนิเมชันวงล้อให้ไปหยุดตรงรางวัลที่เซิร์ฟเวอร์ตอบกลับ → รางวัลเข้าคิวจ่ายรอบสัปดาห์ ผ่านการอนุมัติของผู้ดูแลงบ

จุดที่ห้ามพลาด: **การสุ่มต้องเกิดฝั่งเซิร์ฟเวอร์เท่านั้น** หน้าเว็บมีหน้าที่แค่เล่นแอนิเมชันตามผลที่ได้รับ ถ้าสุ่มฝั่ง browser จะโดนแก้ค่าใน DevTools ได้ทันที

## 5. ข้อกังวลที่ควรพิจารณา (สำคัญ)

**5.1 กฎหมายชิงโชค — ประเด็นใหญ่ที่สุดเพราะเปิดให้บุคคลภายนอกร่วม** การให้ "บุคคลทั่วไป" ร่วมเสี่ยงโชครับรางวัลเป็นเงิน อาจเข้าข่ายการจัดให้มีการแถมพกหรือรางวัลด้วยการเสี่ยงโชค ซึ่งต้องขออนุญาตตามมาตรา 8 พ.ร.บ.การพนัน พ.ศ. 2478 (ยื่นที่ที่ว่าการอำเภอ/สำนักงานเขตพื้นที่) หากจัดโดยไม่มีใบอนุญาตมีโทษทั้งปรับและอาจถึงจำคุก ขอแนะนำอย่างยิ่งให้ปรึกษาฝ่ายกฎหมายก่อนเปิดให้บุคคลภายนอก ทางเลือกที่เสี่ยงน้อยกว่าคือ (ก) เฟสแรกเปิดเฉพาะพนักงานก่อนในฐานะสวัสดิการ/incentive ภายใน แล้วค่อยขยาย หรือ (ข) สำหรับบุคคลภายนอกใช้โครงสร้าง "ค่าแนะนำ (referral fee) อัตราคงที่" ที่ไม่มีองค์ประกอบการเสี่ยงโชค — ทั้งนี้ข้อมูลนี้ไม่ใช่คำแนะนำทางกฎหมาย ควรยืนยันกับนักกฎหมายอีกครั้ง

**5.2 ภาษี** รางวัลจากการชิงโชคโดยทั่วไปต้องหักภาษี ณ ที่จ่าย 5% ส่วนกรณีพนักงาน เงินรางวัลอาจถูกตีความเป็นเงินได้จากการจ้างงานที่ต้องรวมคำนวณภาษีเงินเดือน ควรให้ฝ่ายบัญชี/HR กำหนดแนวปฏิบัติก่อนเปิดใช้ และระบุในเงื่อนไขบนหน้าเว็บว่ารางวัลอาจถูกหักภาษีตามกฎหมาย

**5.3 การโกงและพฤติกรรมบิดเบือน** ความเสี่ยงหลักคือการซอยงานหนึ่งงานเป็นหลายงานเพื่อปั่นสิทธิ์, แจ้งงานปลอม/งานผี, และการสมรู้ร่วมคิดกับผู้ตรวจ แนวป้องกัน: ให้สิทธิ์หลังผ่านการตรวจเท่านั้น (เลือกไว้แล้ว — ดีมาก), นิยาม "งานใหม่" ให้ชัดเป็นลายลักษณ์อักษร, ระบบเช็กงานซ้ำ (ชื่อลูกค้า/เบอร์/สถานที่เดียวกันในช่วงเวลาใกล้กัน), จำกัดสิทธิ์สูงสุดต่อคนต่อวัน (เช่น 5 สิทธิ์) ช่วงแรก, เก็บ audit log ทุกการหมุนและทุกการอนุมัติ, สงวนสิทธิ์เรียกคืนรางวัลหากพบทุจริต, และแยกคนอนุมัติงานออกจากคนที่มีส่วนได้เสีย

**5.4 ความยั่งยืนของแรงจูงใจ** รางวัลเฉลี่ย ~60 บาท/งาน ตื่นเต้นช่วงแรกแต่จะชินชา ควรวางแผนซีซัน เช่น แจ็คพอตพิเศษวันศุกร์, ลีดเดอร์บอร์ดรายเดือน, โบนัสเมื่อทีมทำยอดงานใหม่ถึงเป้า และที่สำคัญ: ระวังผลข้างเคียงที่คนสนใจ "จำนวน" มากกว่า "คุณภาพ" ของงานที่แจ้ง — ควรวัดอัตรางานใหม่ที่ปิดการขาย/ทำจริงได้ควบคู่ไปด้วย เพราะเป้าหมายจริงคือรายได้ ไม่ใช่ยอดแจ้ง

**5.5 PDPA** หากเปิดให้บุคคลภายนอก ต้องเก็บชื่อ เบอร์ บัญชีธนาคารเพื่อโอนรางวัล ต้องมี privacy notice, ขอความยินยอมประกาศรายชื่อผู้โชคดี (แนะนำประกาศแบบปิดบางส่วน เช่น "คุณสมชาย ใ.") และกำหนดระยะเวลาเก็บข้อมูล

**5.6 การเงินและปฏิบัติการ** กำหนดผู้อนุมัติการจ่ายรายสัปดาห์, ทำรายงานกระทบยอด (ผลหมุนในระบบ vs ยอดโอนจริง), ตัดสินใจเรื่องงบเหลือ (แนะนำ: ไม่ทบยอด เพื่อให้ต้นทุนคาดการณ์ได้), และเตรียมคำตอบกรณีข้อพิพาท เช่น ผู้ใช้อ้างว่าหมุนได้ 700 แต่ระบบบันทึก 20 — audit log ฝั่งเซิร์ฟเวอร์คือคำตอบสุดท้าย ควรระบุในเงื่อนไข

**5.7 ROI ที่ควรเช็กกับผู้บริหาร** งบเต็มที่ ~150,000 บาท/เดือน แลกกับงานใหม่ที่เพิ่มขึ้น ~20 งาน/วัน (~600 งาน/เดือน) เท่ากับต้นทุน ~250 บาท/งานใหม่ส่วนเพิ่ม ถ้ามูลค่าเฉลี่ยของงานใหม่ 1 งานสูงกว่านี้ชัดเจนก็คุ้ม ควรกำหนดจุดทบทวนที่ 30/60/90 วัน พร้อมเกณฑ์เลิก/ปรับ เช่น ถ้าสัดส่วนงานใหม่ไม่ขยับใน 60 วัน

## 6. สิ่งที่ต้องเตรียมก่อนพัฒนา

นิยาม "งานใหม่" ที่ชัดเจนและ SLA การตรวจสอบ (แนะนำภายใน 24 ชม. — ถ้าช้ากว่านี้แรงจูงใจจะหายไปมาก), ข้อสรุปด้านกฎหมาย/ภาษีจากฝ่ายที่เกี่ยวข้อง, ช่องทางการจ่ายเงิน (โอนธนาคาร/PromptPay/รวมในเงินเดือน), เงื่อนไขและกติกาฉบับทางการ (T&C), และผู้รับผิดชอบตรวจงาน+อนุมัติจ่าย

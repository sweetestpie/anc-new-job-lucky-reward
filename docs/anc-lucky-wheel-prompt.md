# พรอมพ์สำหรับสั่งพัฒนาฟีเจอร์ Lucky Wheel ใน ANC Portal

> วิธีใช้: คัดลอกพรอมพ์ด้านล่างไปวางให้ Claude Code หรือทีมพัฒนา แล้วเติมข้อมูลในวงเล็บ [...] ให้ตรงกับระบบจริงของคุณก่อนส่ง

---

ฉันต้องการเพิ่มฟีเจอร์ "Lucky Wheel — แจ้งงานใหม่ลุ้นรางวัล" ในระบบ ANC Portal ที่มีอยู่แล้ว

## บริบทของระบบเดิม
- ANC Portal เป็นระบบแจ้งงาน มีงานเข้าประมาณ 200 งาน/วัน
- Stack ปัจจุบัน: [ระบุ เช่น PHP/Laravel + MySQL, Node.js + PostgreSQL, ฯลฯ]
- ระบบล็อกอิน: [ระบุ เช่น พนักงานล็อกอินด้วย employee ID, บุคคลภายนอกใช้เบอร์โทร+OTP]
- ตารางงานเดิม: [ระบุชื่อตาราง/ฟิลด์ เช่น jobs(id, reporter_id, type, status, created_at)]
- งานมีสถานะการตรวจสอบโดยแอดมิน: [ระบุ เช่น status = pending/approved/rejected และฟิลด์ระบุว่าเป็น "งานใหม่"]

## กติกาธุรกิจ (ต้องทำตามนี้เป๊ะ ๆ)
1. เมื่องานถูกแอดมินอนุมัติว่าเป็น "งานใหม่" จริง ให้สร้างสิทธิ์หมุน (spin token) 1 สิทธิ์ให้ผู้แจ้ง ผูกกับ job_id แบบ 1:1 (unique constraint กันสร้างซ้ำเด็ดขาด แม้แอดมินกดอนุมัติซ้ำ)
2. สิทธิ์หมุนมีอายุ 7 วันนับจากวันที่ได้รับ
3. หมุนแล้วได้รางวัลเสมอ รางวัลมี 20 / 50 / 100 / 200 / 300 / 500 / 700 / 1,000 บาท
4. ความน่าจะเป็น (server-side เท่านั้น): 20→50%, 50→28%, 100→14%, 200→5%, 300→2%, 500→0.7%, 700→0.3%
5. แจ็คพอต 1,000 บาท: โอกาส 1.5% ต่อการหมุน แต่จำกัด 1 รางวัล/วันทั้งระบบ — ใช้ atomic operation (เช่น row lock หรือ conditional update) กัน race condition กรณีสองคนหมุนพร้อมกัน
6. งบรางวัลรวมไม่เกิน 5,000 บาท/วัน (ตัดรอบเที่ยงคืนเวลาไทย):
   - ก่อนสุ่มทุกครั้ง เช็กงบคงเหลือของวัน ห้ามจ่ายรางวัลที่ใหญ่กว่างบคงเหลือ
   - ถ้างบคงเหลือ < 300 บาท ให้สุ่มเฉพาะ 20 (60%) / 50 (40%)
   - ถ้างบคงเหลือ < 20 บาท ให้ปิดการหมุนของวันนั้น สิทธิ์ไม่หาย ยกยอดไปหมุนวันถัดไป และแสดงข้อความอธิบายบนหน้าเว็บ
   - งบที่เหลือของแต่ละวันไม่ทบไปวันถัดไป
7. จำกัดสิทธิ์ที่ได้รับสูงสุด 5 สิทธิ์/คน/วัน (config ได้)

## ความปลอดภัย (สำคัญที่สุด)
- การสุ่มรางวัลต้องเกิดฝั่งเซิร์ฟเวอร์ 100% ใช้ CSPRNG (เช่น crypto module) — frontend มีหน้าที่แค่เล่นแอนิเมชันวงล้อให้หยุดตรงผลที่ API ตอบกลับ
- endpoint หมุนต้อง idempotent: รับ spin_token_id, ทำเครื่องหมายใช้แล้วใน transaction เดียวกับการบันทึกผล ถ้า client ยิงซ้ำ (เน็ตหลุด/รีเฟรช) ต้องได้ผลเดิมกลับไป ไม่สุ่มใหม่
- เก็บ audit log ทุกเหตุการณ์: การสร้าง token (ใคร อนุมัติงานไหน เมื่อไหร่), การหมุน (ผล, IP, เวลา), การจ่ายเงิน
- rate limit ที่ endpoint หมุน และตรวจสิทธิ์ว่า token เป็นของผู้ใช้ที่ล็อกอินจริง

## หน้าจอที่ต้องมี
1. **หน้า Lucky Wheel (ผู้ใช้)**: วงล้อ 8 ช่องหมุนได้พร้อมแอนิเมชัน (ดีไซน์ตามไฟล์เดโม่ anc-lucky-wheel-demo.html ที่แนบ), จำนวนสิทธิ์คงเหลือ+วันหมดอายุ, ปุ่มหมุน, ป๊อปอัพแสดงรางวัล, ประวัติรางวัลของตัวเอง+สถานะจ่าย, ฟีดผู้โชคดีล่าสุด (ปิดบังชื่อบางส่วน), สถานะแจ็คพอตวันนี้, กติกาและเงื่อนไข
2. **หน้าแอดมิน**: สรุปงบวันนี้ (ใช้ไป/คงเหลือ/จำนวนหมุน/แจ็คพอตออกหรือยัง), รายการรอจ่ายรายสัปดาห์พร้อมปุ่ม export และ mark as paid, กราฟแนวโน้มสัดส่วนงานใหม่รายวัน (KPI หลักของแคมเปญ: จาก 20% → เป้า 30%), ตารางปรับ config (สัดส่วนรางวัล, งบ/วัน, สิทธิ์สูงสุด/คน/วัน) โดยเก็บประวัติการแก้ไข, รายงานตรวจจับพฤติกรรมผิดปกติ (คนที่แจ้งงานถูกตีตกบ่อย, งานข้อมูลซ้ำกัน)

## Data model ที่แนะนำ
- `spin_tokens`: id, user_id, job_id (unique), status (available/used/expired), created_at, expires_at, used_at
- `spin_results`: id, spin_token_id (unique), user_id, prize_amount, is_jackpot, created_at, ip_address
- `daily_budgets`: date (PK), budget_cap, spent, jackpot_won (boolean) — update แบบ atomic
- `payout_batches` + `payout_items`: รอบจ่ายรายสัปดาห์, สถานะ, ผู้อนุมัติ
- `wheel_config`: สัดส่วนรางวัลและ config ทั้งหมด พร้อม effective_date และประวัติ

## การทดสอบที่ต้องมี
- Unit test การสุ่ม: รัน 100,000 ครั้ง สัดส่วนแต่ละรางวัลต้องใกล้เคียง config (±0.5%)
- Test budget guard: จำลองวันที่มีการหมุน 100+ ครั้ง งบรวมห้ามเกิน 5,000 ไม่ว่ากรณีใด
- Test แจ็คพอต: ยิงหมุนพร้อมกัน (concurrent) หลาย request ต้องมี 1,000 ออกได้แค่ 1 ครั้ง/วัน
- Test idempotency: ยิง request หมุนซ้ำด้วย token เดิม ต้องได้ผลเดิม ไม่กินงบเพิ่ม
- Test การหมดอายุของสิทธิ์ และการยกยอดสิทธิ์เมื่องบหมด

เริ่มจากเสนอ schema และ API design ให้ดูก่อน แล้วค่อย implement ทีละส่วน พร้อมเขียน test ตามที่ระบุ
